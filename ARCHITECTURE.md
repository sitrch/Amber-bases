# Архитектура проекта Amber-bases

## Общее описание
**Amber-bases** — это WPF-приложение (ранее WinForms), которое является модулем более крупного проекта. Приложение предназначено для:
1. Извлечения и отображения данных из базы данных, оформленной в виде книги с листами Excel.
2. Ведения справочников (база данных SQLite `AmberDictionaries.sqlite`) для различных сущностей профилей (поставщики систем, профильные системы, цвета, длины хлыста, типы профилей, применимость и артикулы).

## Структура модулей
Проект разделен на следующие ключевые модули (папки):
- **Core**: Основные классы, интерфейсы, модели данных и конфигурация (включая настройки базы данных SQLite и модели справочников в `Models/Dictionaries`).
- **Services**: Логика работы с источниками данных:
  - Парсинг файлов Excel (через `ExcelDataService`).
  - Работа с базой данных справочников SQLite (через `IDictionaryDataService` и `SqliteDictionaryDataService`).
- **UI**: Элементы интерфейса на базе WPF. Основное окно (`MainWindow`) использует элемент управления `Ribbon` для навигации. Приложение содержит три вкладки Ribbon: "Материалы" (первая), "Загрузчик Excel" и "Справочники". Вкладка "Материалы" отображает таблицу артикулов профилей через `DictionaryEditorView`, открывая её напрямую. Вкладка "Справочники" открывает `DictionaryEditorView` с выбором разных таблиц.

## Структура базы данных и маппинг
База данных (Excel-файл) содержит набор листов, описывающих структуру фасада. Основной таблицей является `РядыРигелейXXX` (где XXX - идентификатор плоскости). Она содержит ссылки на другие таблицы:
- Связи могут быть прямыми (указание ID, например `ТипРядовСтоек` ссылается на `РядыСтоекXXX.тип`).
- Связи могут быть косвенными (значение ячейки содержит имя столбца в целевой таблице, например `ТипРядовЗаполнений` ссылается на столбец с соответствующим именем в таблице `РядыЗаполненийXXX`, при этом индекс строки в целевой таблице определяется как `ТипРядовСтоек - 1`).
Правила маппинга динамически формируются классом `KeyMappingRules` на основе загруженных таблиц.

## Ключевые архитектурные правила
- **Интерфейс Ribbon и меню**: Главное окно приложения использует интерфейс Ribbon. При добавлении новых вкладок (RibbonTab) или других значимых элементов навигации, необходимо добавлять соответствующие пункты в выпадающее главное меню (`RibbonApplicationMenu`) приложения, чтобы обеспечить дублирование навигации.
- **Конфигурация имен**: Базовые настройки, такие как имя файла БД, хранятся в конфигурационных классах (например, `DatabaseConfig`). Ранее использовалась жесткая фиксация имен таблиц, однако для повышения гибкости приложение переведено на динамическое чтение структуры — таблицы и их имена создаются на основе реальных листов внутри Excel-файла.
- **Принципы ООП и DRY**: Крайне важно соблюдать принципы ООП (инкапсуляция, наследование, полиморфизм, абстракция). Обязательно выявлять дублирующийся код или функционал, который может быть переиспользован, и выделять такие функции в отдельные классы или методы (принцип DRY - Don't Repeat Yourself). Не допускается дублирование логики.
- **Документация**: При добавлении нового класса или изменении существующих связей, необходимо обновлять `Level 2` документацию (файлы `.md` в папке `Docs/Classes/...`) и поддерживать актуальность данного `ARCHITECTURE.md`.
- **Комментарии**: Все публичные классы, свойства и методы должны содержать XML-комментарии (`/// <summary>`).
