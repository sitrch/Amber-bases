# Архитектура проекта Amber-bases

## Общее описание
**Amber-bases** — это WPF-приложение (ранее WinForms), которое является модулем более крупного проекта. Приложение предназначено для:
1. Извлечения и отображения данных из базы данных, оформленной в виде книги с листами Excel.
2. Ведения справочников (база данных SQLite `AmberDictionaries.sqlite`) для различных сущностей профилей (поставщики систем, профильные системы, цвета, длины хлыста, типы профилей, применимость и артикулы).

## Структура модулей
Проект разделен на следующие ключевые модули (папки):
- **Core**: Основные классы, интерфейсы, модели данных и конфигурация (включая настройки базы данных SQLite и модели справочников в `Models/Dictionaries`).
- **Services**: Логика работы с источниками данных:
  - Парсинг файлов Excel (через `ExcelDataService`).
  - Работа с базой данных справочников SQLite (через `IDictionaryDataService` и `SqliteDictionaryDataService`).
- **UI**: Элементы интерфейса на базе WPF. Основное окно (`MainWindow`) использует элемент управления `Ribbon` для навигации. Приложение содержит три вкладки Ribbon: "Материалы" (первая), "Загрузчик Excel" и "Справочники". Вкладка "Материалы" отображает таблицу артикулов профилей через `DictionaryEditorView`, открывая её напрямую. Вкладка "Справочники" открывает `DictionaryEditorView` с выбором разных таблиц.

## Структура базы данных и маппинг
База данных (Excel-файл) содержит набор листов, описывающих структуру фасада. Основной таблицей является `РядыРигелейXXX` (где XXX - идентификатор плоскости). Она содержит ссылки на другие таблицы:
- Связи могут быть прямыми (указание ID, например `ТипРядовСтоек` ссылается на `РядыСтоекXXX.тип`).
- Связи могут быть косвенными (значение ячейки содержит имя столбца в целевой таблице, например `ТипРядовЗаполнений` ссылается на столбец с соответствующим именем в таблице `РядыЗаполненийXXX`, при этом индекс строки в целевой таблице определяется как `ТипРядовСтоек - 1`).
Правила маппинга динамически формируются классом `KeyMappingRules` на основе загруженных таблиц.

## Единообразное отображение таблиц
Все таблицы в приложении используют единый компонент `DictionaryTableControl` (WPF `UserControl`) для просмотра и редактирования данных. Принцип единообразия обеспечивается следующим:

1. **Авто-генерация столбцов** — `DictionaryTableControl.SetupColumns()` через рефлексию создаёт столбцы `DataGrid` на основе свойств модели (тип `BaseDictionaryModel`). Технические поля (`Id`, `Position`, `Info`) и навигационные коллекции исключаются.
2. **FK-поля (внешние ключи)** — Свойства с суффиксом `Id` автоматически отображаются как `DataGridComboBoxColumn` с lookup-данными из словаря `allCollections` (например, `ManufacturerId` → `SystemProvider.Name`). Словарь маппингов FK определён в `GetParentEntityType()` и дублируется в `UpdateContextMenuForDataGrid()` и `OpenParentTableFromCell()`.
3. **Русифицированные заголовки** — `GetPropertyDisplayName()` содержит единый словарь перевода имён свойств в читаемые заголовки (напр. `CutWisibleWidth` → "Ширина отображения").
4. **Отображение типов** — `GetDisplayName()` содержит единый словарь перевода имён типов в заголовки таблиц (напр. `ProfileArticle` → "Артикулы").
5. **Контекстное меню (правый клик)** — у всех таблиц одинаковое контекстное меню:
   - **Добавить строку** — создаёт экземпляр через `Activator.CreateInstance` и добавляет в коллекцию.
   - **Редактировать** — открывает `ModelEditorWindow` для детального редактирования. ВАЖНО: в редактор передаётся реальный объект модели из `_collection`, а не `DataRowView` из `TrackingTable`. Для конвертации используется `GetSourceItemFromDataGrid()`.
   - **Удалить строку** — удаляет выделенные строки с подтверждением (`MessageBox` Yes/No).
   - **Копировать** — копирует выделенные ячейки в TSV-формате (Tab-Separated Values) с экранированием кавычек.
   - **Вставить из Excel** — парсит TSV из буфера обмена, обновляет существующие строки или создаёт новые.
   - **Открыть таблицу "[Родительская]"** — динамически добавляется при клике на FK-ячейку (столбец ending with `Id`).
6. **Горячие клавиши** — `Ctrl+Z`/`Ctrl+Y` (Undo/Redo), `Ctrl+C`/`Ctrl+V` (Копировать/Вставить), `Delete` (Удалить), `Ctrl+D` (Дублировать), `Enter` (Открыть редактор).
7. **FK-навигация** — все FK-поля (заканчиваются на `Id`) отображаются через `DataGridComboBoxColumn` с lookup-данными. Тип родительской сущности определяется через `GetParentEntityType(PropertyInfo)`, который проверяет navigation property и использует единый словарь `typeMappings`. Этот метод — единственный источник маппинга FK; дублирование словарей в других методах запрещено.
8. **Конвертация типов при вставке** — `int`/`int?` (пустое → `0`/`null`), `double`/`double?` (пустое → `0.0`/`null`, поддержка `.` и `,`), FK-поля (числовой ID напрямую, строковое имя → lookup по `Name`).
9. **Undo/Redo** — после любого редактирования через `ModelEditorWindow` вызывается `_actionTracker.DetectAndRecordChanges()` для фиксации изменений. Без этого Undo/Redo не будет работать.
10. **Миграции БД** — при изменении структуры модели (добавление/удаление колонок) SQL-миграции размещаются в `Migrations/` с форматом `YYYYMMDD_HHMM_Description.sql`. Одновременно с кодом приложения обновляется `SqliteDictionaryDataService.InitializeDatabase()` для автоматического применения `ALTER TABLE` при старте.

## Ключевые архитектурные правила
- **Интерфейс Ribbon и меню**: Главное окно приложения использует интерфейс Ribbon. При добавлении новых вкладок (RibbonTab) или других значимых элементов навигации, необходимо добавлять соответствующие пункты в выпадающее главное меню (`RibbonApplicationMenu`) приложения, чтобы обеспечить дублирование навигации.
- **Конфигурация имен**: Базовые настройки, такие как имя файла БД, хранятся в конфигурационных классах (например, `DatabaseConfig`). Ранее использовалась жесткая фиксация имен таблиц, однако для повышения гибкости приложение переведено на динамическое чтение структуры — таблицы и их имена создаются на основе реальных листов внутри Excel-файла.
- **Принципы ООП и DRY**: Крайне важно соблюдать принципы ООП (инкапсуляция, наследование, полиморфизм, абстракция). Обязательно выявлять дублирующийся код или функционал, который может быть переиспользован, и выделять такие функции в отдельные классы или методы (принцип DRY - Don't Repeat Yourself). Не допускается дублирование логики.
- **Документация**: При добавлении нового класса или изменении существующих связей, необходимо обновлять `Level 2` документацию (файлы `.md` в папке `Docs/Classes/...`) и поддерживать актуальность данного `ARCHITECTURE.md`.
- **Комментарии**: Все публичные классы, свойства и методы должны содержать XML-комментарии (`/// <summary>`).
- **Структура классов моделей**: Все классы моделей справочников (наследники `BaseDictionaryModel`) должны использовать регионы для логической группировки свойств. Обязательная структура:
  1. `#region Основные свойства` — основные бизнес-поля модели (название, описание, код и т.д.)
  2. `#region Служебные свойства` — поля внешних ключей с суффиксом `Id` (например, `ManufacturerId`, `SystemId`, `ColorId`)
  3. `#region Навигационные свойства` — ссылки на связанные объекты (например, `Manufacturer`, `System`, `Color`)
  
  Навигационные и служебные свойства должны быть сгруппированы в конце класса после основных свойств. Такое разделение улучшает читаемость кода и автоматическую генерацию UI.
