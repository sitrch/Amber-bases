# Архитектура проекта Amber-bases

## Общее описание
**Amber-bases** — это WPF-приложение на базе элемента управления Ribbon. Приложение предназначено для:
1. Извлечения и отображения данных из Excel-файлов (база данных в виде книги с листами).
2. Ведения справочников (база данных SQLite `AmberDictionaries.sqlite`) для различных сущностей профилей:
   - Поставщики систем (SystemProvider)
   - Профильные системы (ProfileSystem)
   - Цвета (Color)
   - Стандартные длины прутков (StandartBarLength)
   - Типы профилей (ProfileType)
   - Применимость (Applicability)
   - Артикулы профилей (ProfileArticle)
   - Типы покрытий (CoatingType)
   - C-профили (CProfile)

## Структура модулей
Проект разделен на следующие ключевые модули (папки):
- **Core**: Основные классы, интерфейсы, модели данных и конфигурация:
  - `Models/Dictionaries/` — модели справочников (наследники `BaseDictionaryModel`)
  - `DatabaseConfig.cs` — конфигурация БД
  - `Builders/SqlBuilder.cs` — генератор SQL
  - `Interfaces/IInitControl.cs` — интерфейс инициализации
  - `Repositories/` — базовый репозиторий (пока не используется)
- **Services**: Логика работы с источниками данных:
  - Парсинг файлов Excel (через `IExcelDataService` / `ExcelDataService`)
  - Работа с базой данных справочников SQLite (через `IDictionaryDataService` / `SqliteDictionaryDataService`)
- **UI**: Элементы интерфейса на базе WPF:
  - `MainWindow` — главное окно с Ribbon-навигацией
  - `DictionaryEditorView` — панель редактора справочников
  - `DictionaryTableControl` — универсальный контрол таблицы
  - `ModelEditorWindow` — модальное окно редактирования записи
  - `TableViewWindow` — модальное окно просмотра таблицы
  - `Tracking/` — отслеживание изменений (Undo/Redo)

## Навигация Ribbon
Главное окно (`MainWindow`) использует элемент управления `Ribbon`. Три вкладки:
1. **Материалы** — отображает таблицу артикулов профилей (ProfileArticle)
2. **Загрузчик Excel** — загрузка и сохранение Excel-данных в SQLite
3. **Справочники** — полный интерфейс редактирования справочников

## База данных справочников
Приложение использует SQLite-базу `AmberDictionaries.sqlite` для хранения справочников. Инициализация и миграции выполняются в `SqliteDictionaryDataService.InitializeDatabase()`:
- Создание таблиц на основе рефлексии моделей (через `SqlBuilder<T>`)
- Добавление недостающих колонок через `ALTER TABLE` для обратной совместимости

### Таблицы справочников
| Модель | Таблица | Описание |

## Единообразное отображение таблиц
Все таблицы в приложении используют единый компонент `DictionaryTableControl` (WPF `UserControl`) для просмотра и редактирования данных. Принцип единообразия обеспечивается следующим:

1. **Авто-генерация столбцов** — `DictionaryTableControl.SetupColumns()` через рефлексию создаёт столбцы `DataGrid` на основе свойств модели (тип `BaseDictionaryModel`). Технические поля (`Id`, `Position`, `Info`) и навигационные коллекции исключаются.
2. **FK-поля (внешние ключи)** — Свойства с суффиксом `Id` автоматически отображаются как `DataGridComboBoxColumn` с lookup-данными из словаря `allCollections` (например, `ManufacturerId` → `SystemProvider.Name`). Маппинг FK определён в методе `GetParentEntityType(PropertyInfo)`.
3. **Русифицированные заголовки** — `DisplayNameProvider.GetPropertyName()` содержит единый словарь перевода имён свойств в читаемые заголовки (напр. `CutWisibleWidth` → "Ширина отображения").
4. **Контекстное меню (правый клик)** — у всех таблиц одинаковое контекстное меню:
   - **Добавить строку** — создаёт экземпляр через `Activator.CreateInstance` и добавляет в коллекцию.
   - **Редактировать** — открывает `ModelEditorWindow` для детального редактирования. ВАЖНО: в редактор передаётся реальный объект модели из `_collection`, а не `DataRowView` из `TrackingTable`. Для конвертации используется `GetSourceItemFromDataGrid()`.
   - **Удалить строку** — удаляет выделенные строки с подтверждением (`MessageBox` Yes/No).
   - **Копировать** — копирует выделенные ячейки в TSV-формате (Tab-Separated Values) с экранированием кавычек.
   - **Вставить из Excel** — парсит TSV из буфера обмена, обновляет существующие строки или создаёт новые.
   - **Открыть таблицу "[Родительская]"** — динамически добавляется при клике на FK-ячейку.
5. **Горячие клавиши** — `Ctrl+Z`/`Ctrl+Y` (Undo/Redo), `Ctrl+C`/`Ctrl+V` (Копировать/Вставить), `Delete` (Удалить), `Ctrl+D` (Дублировать), `Enter` (Открыть редактор).
6. **FK-навигация** — все FK-поля (заканчиваются на `Id`) отображаются через `DataGridComboBoxColumn` с lookup-данными. Тип родительской сущности определяется через `GetParentEntityType(PropertyInfo)`.
7. **Конвертация типов при вставке** — `int`/`int?` (пустое → `0`/`null`), `double`/`double?` (пустое → `0.0`/`null`, поддержка `.` и `,`), FK-поля (числовой ID напрямую, строковое имя → lookup по `Name`).
8. **Undo/Redo** — реализуется через `EditActionTracker`. После любого редактирования (прямо в ячейке или через `ModelEditorWindow`) вызывается `_actionTracker.DetectAndRecordChanges()` для фиксации изменений. Без этого Undo/Redo не будет работать.
9. **Миграции БД** — при изменении структуры модели (добавление/удаление колонок) SQL-миграции размещаются в `Migrations/` с форматом `YYYYMMDD_HHMM_Description.sql`. ## Ключевые архитектурные правила
- **Интерфейс Ribbon и меню**: Главное окно приложения использует интерфейс Ribbon. При добавлении новых вкладок (RibbonTab) или других значимых элементов навигации, необходимо добавлять соответствующие пункты в выпадающее главное меню (`RibbonApplicationMenu`) приложения, чтобы обеспечить дублирование навигации.
- **Конфигурация имен**: Базовые настройки, такие как имя файла БД, хранятся в конфигурационных классах (например, `DatabaseConfig`). Ранее использовалась жесткая фиксация имен таблиц, однако для повышения гибкости приложение переведено на динамическое чтение структуры — таблицы и их имена создаются на основе реальных листов внутри Excel-файла.
- **Принципы ООП и DRY**: Крайне важно соблюдать принципы ООП (инкапсуляция, наследование, полиморфизм, абстракция). Обязательно выявлять дублирующийся код или функционал, который может быть переиспользован, и выделять такие функции в отдельные классы или методы (принцип DRY - Don't Repeat Yourself). Не допускается дублирование логики.
- **Документация**: При добавлении нового класса или изменении существующих связей, необходимо обновлять `Level 2` документацию (файлы `.md` в папке `Docs/Classes/...`) и поддерживать актуальность данного `ARCHITECTURE.md`.
- **Комментарии**: Все публичные классы, свойства и методы должны содержать XML-комментарии (`/// <summary>`).
- **Структура классов моделей**: Все классы моделей справочников (наследники `BaseDictionaryModel`) должны использовать регионы для логической группировки свойств. Обязательная структура:
  1. `#region Основные свойства` — основные бизнес-поля модели (название, описание, код и т.д.)
  2. `#region Служебные свойства` — поля внешних ключей с суффиксом `Id` (например, `ManufacturerId`, `SystemId`, `ColorId`)
  3. `#region Навигационные свойства` — ссылки на связанные объекты (например, `Manufacturer`, `System`, `Color`)
  
  Навигационные и служебные свойства должны быть сгруппированы в конце класса после основных свойств. Такое разделение улучшает читаемость кода и автоматическую генерацию UI.
